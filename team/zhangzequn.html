PROJECT: Delivery

Overview

Delivery is a desktop application designed for restaurants who want to maintain their own system to keep track customers’ orders. It is optimized for those who prefer to work with a Command Line Interface (CLI) while still having the benefits of a Graphical User Interface (GUI). It is written in Java, and has about 10 kLoC.

Menu feature

Current Implementation

All the dishes can be created as an object and then be stored in an additional local database. The data can be kept in a local XmlMenuStorage file.

The data file consists of all the available dishes and drinks provided by the restaurant. Customers can only order those provided. In the case when anything which is not provided by the restaurant but is ordered by customers, exceptions will be thrown. Notification will be given to inform the user that customers should not order this particular dish or drink.

Users will be able to update the data file by adding or deleting dishes objects inside the database.

Design Considerations

Aspect: Restriction for ordering

  • Alternative 1 (current choice): Allow user to input the names for dishes and drinks

    • Pros: user need not to have a index reference.

    • Cons: It could be time-consuming for typing all the names. There might be typos which make dishes can not be recognised.

  • Alternative 2: Allow user to input the indexes for dishes and drinks

    • Pros: Easy to implement.

    • Cons: User needs a index reference. Work needs to be done while maintaining the index reference.

Aspect: Maintaining the menu

  • Alternative 1 (proposed): Use lazy deletion.

    • Pros: Easy to implement.

    • Cons: It may take a lot of memory space if there are a lot of dishes updated.

  • Alternative 2: Delete the dish object.

    • Pros: Save memory space

    • Cons: While adding back those pre-existing dishes, more work need to be done by programmer.

Aspect: Data structure to store menu

  • Alternative 1 (current choice): Use a HashMap. String will be the key, Dishes will be the value.

    • Pros: Efficient to search for a given key and decide whether the dish is available.

    • Cons: User may key in wrongly for names of dishes.

  • Alternative 2: Use a HashMap. Dishes will be the key, prices will be the value.

    • Pros: It is easy to retrieve the price for each dishes.

    • Cons: It could be time-consuming for typing all the names. There might be typos which make dishes can not be recognised.
